I asked an AI collaborator to build me a localization fit-checker for my game,
and it built a nice visual that measured whether my strings fit in an invented map.
The math was right, but the thing it was measuring against was its best guess.
What started out as what I thought was some quick math, turned out to be a story about
the one skill that actually matters when you work with AI: knowing which
plausible-looking output to distrust. Let's look at how I applied creative problem-solving 
to build something better. The messy backlog I still haven't cleaned up, is the correct 
cost of a game very much at the prototype stage still finding where its fun is.

<Breakout kind="takeaway">
The dangerous output of an AI agent isn't the wrong answer. It's the intuitive, 
plausible one. My rigor wasn't in the prompt; it was in refusing to believe the
meter until it could show me its receipt.
</Breakout>

## The meter that looked right

Familiar is a TTRPG-inspired roguelike where you play the adventurer's familiar:
you nudge a hero who already has their own plans. Influence actions can lower a
check's difficulty, improve the odds, or change what the hero does outright, and
those moments want their own voice, their own font, their own authored lines.
Some of those lines exist already. (Hold onto that. This weekend's audit is how I
learned/remembered the game has no way to show them yet.)

That premise makes localization a first-class problem, not a ship-it-later
afterthought. Text is dynamic on every axis at once: keys resolved at runtime,
tokens substituted, fragments drawn at random, boxes instanced from prefabs,
3+ font "voices" carrying who's speaking. On mobile, an overflowed button or a
stray `TODO` string in a second language leaves no screen real estate to hide. I'd
been careful about it from early on, no player-facing text in content JSON, every
display string keyed to a sidecar table. Over a decade building games, some of it spent
shoulder to shoulder with external studios trying to get things shippable, teaches
you that some problems punish the people who defer it disproportionately.

And I'd still call "I designed it right" an overstatement, because here's what the
game actually looked like while I was busy finding the fun:

So over a long weekend, in the gaps between family time, I set out to use AI to build recepits
for what felt like a small question: *are my strings the right length?* I had already built a content dashboard 
and now asked for a length-checker. The first character meter rendered, and it looked like it worked.

## I asked where the mapping came from. Claude told me it guessed.

I got the little hit of excitement, my pain point solved, and then I asked the
one question that unravels everything: how does it know *that* string lands in
*that* box?

It didn't. It had pattern-matched a plausible mapping and measured fit against it.
The geometry was honest, the box really did hold about ten characters, but the
claim that a given sentence lived in that box was invented. A fit check against a
guessed mapping can be confidently, uselessly wrong: it might be measuring the
right string against the wrong box entirely.

<PullQuote>
A meter that measures a guess isn't a measurement- it's a goose chase.
</PullQuote>

## Guessing smarter doesn't work

My first fix was the obvious one: ground the mapping in truth by having the AI
trace the code. It built a strategy that assumed **static** localization keys, and
that fell apart immediately, because my keys aren't static. They're assembled at
runtime from tokens, in-game variables, and content ids. Tracing source for a string 
that doesn't exist as a literal until the game builds it is a dead end.

The reframe came when I noticed the fit bug had siblings. Hardcoded strings, an
enum printed straight to a label with `.ToString()`, baked placeholder text in a
prefab, were the *same disease*: text reaching the player without going through my
localization system at all. Little shortcuts that come from using AI agents to build a 
prototype quickly and pivoting constantly. I stopped chasing the one symptom and went 
looking for a root-cause solution to the whole class.

<Breakout kind="lesson">
You can't derive runtime truth from static a source. Static tracing was doomed
twice over here: the keys resolve at runtime, and the boxes are prefab-instanced,
so a shop row or an item card doesn't even exist in the scene file to inspect.
There was nothing to read.
</Breakout>

And the guess wasn't just unfounded, it was actively hiding things. The first few
passes were very messy. Following the localization keys to the logical place in-game 
also failed; I had inadvertantly allowed my team to define strings that served the 
same purpose. My UI localization file had different copy explaining the influence 
actions available to the player than the per-familiar flavored explanations (when I 
built the content pipeline, I figured a wolf would observe differently than an owl - 
smelling things versus a birds eye view). Putting the content pipeline before the game was 
right for this kind of exploratory project, but obviously pipeline engineers disagree 
with UI/UX developers all the time. I just happened to be both and also neither.

## Stop guessing about the project; record what happens at runtime

Here's the aha. There is exactly one instant when the key and the box are both in
hand: the moment you assign the text. So I made that a single choke-point
function and had it keep a receipt. I built a runtime ledger to act as the source 
of truth for which keys got localized to what boxes with what settings.

The one canonical setter resolves the key, substitutes tokens, assigns the box,
and, in the editor only, records `key → box → value` to a ledger with zero
inference. That ledger, built out of the running game, sets the pattern. It
costs nothing in the shipped build; the instrumentation exists only where dev
and QA happen, a fine place to take a performance hit.

<Insight sells="root-cause instrumentation">
The move that generalizes past this game: when truth is impossible to infer
everywhere, find the single point where it exists for free, and record it there.
I didn't teach a tool to guess better. I found the one line of code that already
knew the answer and promoted it to accountant.
</Insight>

I built the ledger to accumulate across play sessions and clear only on purpose,
because one playthrough rarely touches every scene. The commit history from that
weekend is a run of fast checkpoints, not thrashing: I knew the shape I wanted.
What I was building, under all the machinery, was a cohesive strategy to enforce 
my "localization as a first-class citizen" principple.

Then I put it where I'd trip over it. The ledger lives in a tab beside Scene and
Simulator, so every time I play the game I am one click from the receipt.

Don't hide the mess. Make the reconciliation between intent and what's actually happening something I can't ignore.
</PullQuote>

## The ledger has its own blind spot

Then the ledger tried to fool me the same way the meter had, and this time I saw
it coming. By construction, a ledger only records text that went *through* the
system. It is perfectly blind to exactly the class of issues I cared about most: 
baked prefab text, the `enum.ToString()`, the raw literal. A clean-looking ledger can
still be missing every un-localized string in the game.

So I built its complement: a census that walks every text box that actually gets 
rendered, localized or not. The set difference is the whole point.

```
census   = every TMP box that displayed real copy   (what the player saw)
ledger   = every TMP box localized in-game     (what went through the system)

census ∖ ledger = copy that rendered without being localized
```

Immediately it earned its keep: my plot-resource grant text, the familiar
overhearing a rumor, had no in-game home. The AI wanted to suppress the finding, because 
there were *so many* of them, surely the check was miscalibrated. I stopped it. I
threw a mockup dialogue into the scene just to see whether the ledger would catch
it, and went to go live my weekend. I came back to find the placeholder title
missing and no keys defined for any of the rumor types I'd already authored. The tool 
wasn't catching me being wrong, but showing me where the game wasn't built yet.

<Breakout kind="principle">
Don't turn the metric green. The AI's instinct when a check screams was to assume
the check was broken and quiet it, or infer why the rule was wrong (no schema contracts 
for localization tokens was an early choice I'd made to support iteration). The check was 
the only thing telling the truth. A good dashboard cannot hide the signal, even when the signal 
appears overwhelming, like it did to the AI agent iterating on it.
</Breakout>

## One string vouches for a thousand

Because localization keys are tokenized and dynamic, showing a *single* rumor in-game
proved the box for the *whole family*. The code already routes every string in a
family to the same box regardless of rumor id, so I let the code assert the
route and the reconciliation confirm, refute, or leave that family of keys unproven. 
Every mapping now carries its provenance: how do I actually know this?

point: this got built in the margins of a real weekend, and the ledger reflects
exactly where I left it as I enjoyed my time with my family.

That's the ladder the whole weekend was climbing: a naive guess gets replaced by
code-truth, which gets confirmed by observation. As I playtest more, I see strings 
move from "codepath confirmed" to "runtime verified" and know exactly why, when and how.

## The question I started with, finally answered

With mappings that were finally true, I could do the small thing I set out to do
on day one: find out which strings overflow. In the rumor family, almost all of
them are at risk against the placeholder box I dropped in.

These overflows had been invisible the whole time, because the original hand guess
had pointed the family at a scrolling box that never checks length. Fixing the
*mapping* is what made the overflow findable. The measurement was never the hard
part: it required grounding the idea of what is an acceptable assertion.

## A beautiful disaster, and I'm leaving it that way (for now)

I could make every number on this dashboard go green tonight. Rewrite the prefab
metadata to fix the font-voice warnings, migrate the hardcoded literals into the
string tables, wire the unmapped strings to keys that mostly already exist. It's
all mechanical but I haven't, and that's the whole crux of why I chose a dashboard at all. 
This is very much a prototype that needs time to breathe and no amount of AI-assisted power 
up to development speed will tell me which of these numbers should be green, and which 
aren't worth chasing at all because that's kind of the fun part anyway. Building a 
dashboard to set burndown goals is tempting in a lot of ways, but at this stage I 
want to build dashboards to make sure I have the data to support my decisions.

Some of the mess is here because I'm still tuning my voice rules, and I needed help 
looking at which decisions to stand on, which to automate away, and which to think about.
Some of it is here because the moment I could see the whole content system clearly, I got
curious about the next thing and wandered off to explore it. Rushing to clean it
up would repeat the original mistake: freezing a contract, making rules for copy a game
didn't actually support, vague voice rules not documented in a way the agents could adhere to, 
on a game that hasn't finished deciding what it is. The influence lines I
mentioned at the top, the ones with nowhere to render yet? Deleted on purpose, not silenced.

The instrument exists so I stay free to chase the fun, not just so the dashboard turns
green (yet). If I wanted them all to be green, I would have built skills, robust editor tools and tests.
This game is joyfully in prototype stage, and the dashboard reflects that
honestly. For now, I'm just as excited to explore whether to ship a Unity
plugin with polished, deterministic versions of these techniques to help devs course-correct after things invevitably get messy. 


Isn't it fun what you learn when you let curiosity take the wheel every so often?
