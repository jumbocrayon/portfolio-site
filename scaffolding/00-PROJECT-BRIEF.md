# Portfolio Project Brief — "The Living Testbed"

## What this is
A personal portfolio website that is **not a résumé** — it's a living testbed where I run my own theories on interactive media in public. The site doesn't *describe* how I work; it *demonstrates* it by being built the same way I'd run an engagement.

**Important scope note:** This project is the **portfolio site scaffolding** — the container, the shell, the structure. It is **separate from the dev blog** (the dev blog is a downstream content stream that will live inside or alongside this shell, but is not what we're building first). Right now we are prototyping the *scaffolding fast*.

## Who I am (positioning)
Executive-level partner in interactive media who leads with curiosity and earns trust through self-awareness and results. I think in systems but communicate in human terms. I take risks when it's cheap to learn and build strategy before the stakes get high. Clever without being arrogant, compassionate without being soft.

**One-line thesis the whole site argues:**
> "Applied curiosity + caring about how creative groups use available resources = art meets science in unimaginably cool places, and I curate those spaces. I set technical foundations where curiosity and magic can later happen."

## Who it's for
Prospective fractional clients and creative/technical partners — people deciding whether to work with me. The site should let them feel, in order:
1. **Whimsy in service of the work** (the hook) — playfulness that visibly moves a project forward, not decoration.
2. **A feel for partnering with me** (the character read) — what I bring, made legible through the values I won't negotiate to get there.
3. **Systems-level technical depth** (the proof) — an in-depth deep dive that builds trust and shows technical depth + analysis actually making something work.

Emotional → relational → credibility, in that sequence. Whimsy gets them in the door; values show them who they'd work with; depth closes the trust gap.

## The design principle (non-negotiable)
**The medium is the message.** The site's *structure* mirrors my *operating values* because they're the same commitments: I lead with whimsy because I value play, I go deep because I value depth, I build trust through values because relationship is how I work. The site should demonstrate how I work by being built that way — not by claiming it.

## Strategic purpose (why it exists)
- **Outward proof:** demonstrate I can set a technical foundation for curiosity and magic — so I attract work I don't have to argue the premium price for. The demo does the convincing.
- **Inward diagnostic:** the site is also a dual-purpose instrument. Engagement analytics are a leading indicator of where to invest my time. ("If a prospective partner won't even listen to the dev blog, the game probably needs more love before it's fun for anyone else.")

## Content that will eventually live here (NOT all built now — this is the map)
1. **Centerpiece — comic-vignette engine:** human-defined art style + real-time AI animation of a single told moment into a cute, customizable, shareable animated comic. The flagship "art meets science made touchable" demo. (Originates as a Familiar feature: a playthrough's story → shareable animated vignette.)
2. **Dev blog (separate stream):** ~10-min Familiar dev-blog episodes; animating a moment from an episode doubles as first-pass feature code. One build wearing three hats.
3. **Pre-mortem worksheet (satellite demo):** interactive "villain of the week" pre-mortem tool from PS5-launch expertise.
4. **Kids' "let's tell a story" app (the spark test):** lowest-touch test of the magic atom, in hand-drawn art true to me — the cheapest way to prove the centerpiece works.

## What to build FIRST (this project's actual job)
The **scaffolding**: a fast, flexible site shell that can host a growing collection of interactive demos, each framed by the whimsy→depth→values spine, with slots for the centerpiece, the dev blog stream, and satellite demos. Prioritize speed-to-prototype and the ability to drop in new interactive pieces without rearchitecting.

## Constraints / context
- This is a low-overcommitment, curiosity-led project that should compound gently — not a heavyweight build. Favor simple, fast, fun-to-iterate tooling over enterprise scaffolding.
- I can draw "well enough" — hand-made, true-to-me aesthetic is a feature, not a bug. The visual language should feel personal and a little handmade, not corporate-polished.
